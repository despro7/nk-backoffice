# План реалізації offline-first архітектури збірки замовлень

## 🔍 Поточна ситуація

### Проблеми
1. **Блокуючі модальні вікна помилок** при втраті інтернет-з'єднання:
   - Під час збірки замовлення користувач може втратити інтернет
   - Спроба оновити статус до "Готове до відправки" блокується помилкою мережі
   - Показується блокуюче модальне вікно з помилкою, що перериває роботу

2. **Втрата прогресу збірки** при офлайн-сценаріях:
   - Чекліст збірки не зберігається локально
   - При перезавантаженні сторінки або поверненні до замовлення прогрес втрачається
   - Користувач (наприклад, Аліна) змушений починати збірку заново

3. **Відсутність оптимістичних оновлень**:
   - Всі зміни статусу чекають підтвердження від сервера
   - UI блокується при мережевих проблемах
   - Немає візуального фідбеку про локальні зміни

4. **Відсутність механізмів відновлення**:
   - Неповні замовлення не мають індикації про незавершену збірку
   - Немає можливості продовжити збірку з того місця, де вона була перервана

## 📋 План реалізації

### **ФАЗА 1: Локальне збереження прогресу збірки** (2-3 дні)

#### 1.1 Розширення компонента `OrderChecklist`
**Файл**: `client/components/OrderChecklist.tsx`

**Додати функціонал**:
- Автоматичне збереження стану чекліста в localStorage після кожної успішної дії
- Відновлення стану при завантаженні замовлення
- Індикація відновленого прогресу з попередньої сесії

**Ключові методи**:
```typescript
// Автозбереження після зміни елемента
const saveChecklistProgress = useCallback((itemId: string, completed: boolean) => {
  const key = `order-checklist-${order.externalId}`;
  const progress = JSON.parse(localStorage.getItem(key) || '{}');
  progress[itemId] = { completed, timestamp: Date.now() };
  localStorage.setItem(key, JSON.stringify(progress));
}, [order.externalId]);

// Відновлення прогресу при завантаженні
const loadChecklistProgress = useCallback(() => {
  const key = `order-checklist-${order.externalId}`;
  const saved = localStorage.getItem(key);
  if (saved) {
    const progress = JSON.parse(saved);
    // Застосувати збережений стан до чекліста
    return progress;
  }
  return {};
}, [order.externalId]);
```

#### 1.2 Модальне вікно відновлення прогресу
**Файл**: `client/components/modals/ChecklistRecoveryModal.tsx`

**Функціонал**:
- Показується при завантаженні замовлення з незавершеним прогресом
- Відображає статистику: "Знайдено 3/5 завершених елементів з попередньої сесії"
- Кнопки: "Продовжити збірку" / "Почати заново" / "Показати деталі"

#### 1.3 Очищення застарілого localStorage
**Файл**: `client/services/CleanupService.ts`

**Логіка очищення**:
- Видаляти записи старші 7 днів
- Очищати при успішному завершенні замовлення
- Обмежувати розмір localStorage (максимум 50MB)

### **ФАЗА 2: Оптимістичні оновлення UI** (1-2 дні)

#### 2.1 Модифікація `useOrderNavigation` хука
**Файл**: `client/hooks/useOrderNavigation.ts`

**Додати оптимістичні оновлення**:
```typescript
const updateCurrentOrderStatusToReady = useCallback(async () => {
  // Оптимістичне оновлення UI
  const optimisticUpdate = { status: '3', statusText: 'Готове до відправки' };
  onUpdateOrderStatus(optimisticUpdate);

  try {
    // Фонове оновлення на сервері
    await apiCall('/api/orders/' + order.externalId + '/status', 'PUT', {
      status: '3',
      statusText: 'Готове до відправки'
    });

		// Блокування webhook SalesDrive щоб уникнути зайвих змін статусу замовлення через відлуння webhook
		// TODO: Треба додати код

    // Підтвердження успішного оновлення
    showSuccessToast('Статус оновлено успішно');
  } catch (error) {
    // Відкат оптимістичного оновлення
    onUpdateOrderStatus(order.originalStatus);
    showErrorToast('Не вдалося оновити статус. Спробуйте ще раз.');
  }
}, [order, onUpdateOrderStatus]);
```

#### 2.2 Неблокуючі повідомлення замість модальних вікон
**Використання існуючого `ToastService`**:
- Замість блокуючих модальних вікон показувати toast повідомлення
- Дозволити користувачу продовжувати роботу
- Додати кнопку "Повторити" в toast для повторних спроб

### **ФАЗА 3: Фонова синхронізація** (2-3 дні)

#### 3.1 Service Worker для фонової синхронізації
**Файл**: `public/sw.js`

**Функціонал**:
- Перехоплення невдалих мережевих запитів
- Збереження їх в IndexedDB для подальшої синхронізації
- Автоматична повторна відправка при відновленні з'єднання

#### 3.2 Серверний механізм повторних спроб
**Файл**: `server/services/SyncRetryService.ts`

**Логіка**:
- Cron job що перевіряє невдалі синхронізації кожні 5 хвилин
- Повторна відправка статусів до SalesDrive
- Логування в `meta_logs` для адміністраторів
- Експоненціальна затримка між спробами (1min → 5min → 15min → 1год)

#### 3.3 Індикація статусу синхронізації
**Файл**: `client/components/SyncStatusIndicator.tsx`

**UI елементи**:
- Іконка біля номера замовлення показує статус синхронізації
- "✅ Синхронізовано" / "⏳ Синхронізується" / "❌ Помилка синхронізації"
- Tooltip з деталями останньої спроби

### **ФАЗА 4: Використання існуючої інфраструктури** (1 день)

#### 4.1 Інтеграція з `useServerStatus` хуком
**Переваги існуючого рішення**:
- Вже перевіряє `/api/health` кожні 15 секунд
- Надає булевий `isOnline` стан
- Має `ServerStatusContext` для глобального доступу

**Використання**:
```typescript
const { isOnline } = useServerStatus();

// Показувати різні UI залежно від статусу з'єднання
{isOnline ? (
  <Button onClick={updateStatus}>Оновити статус</Button>
) : (
  <Button disabled>Оновити статус (офлайн)</Button>
)}
```

#### 4.2 Не створювати дублюючий `useOnlineStatus` хук
**Причина**: Існуючий `useServerStatus` повністю покриває потреби офлайн-детекції

## 🎯 Очікуваний результат

### Для користувачів (складських працівників)
1. **Безперервна робота** при втраті інтернет-з'єднання
2. **Автоматичне збереження** прогресу збірки
3. **Відновлення роботи** з того місця, де вона була перервана
4. **Оптимістичний UI** що не блокується мережевими проблемами
5. **Неінтрузивні повідомлення** замість блокуючих модальних вікон

### Для адміністраторів
1. **Видимість офлайн-активності** через логи та індикатори
2. **Автоматична синхронізація** при відновленні з'єднання
3. **Сповіщення про проблеми** синхронізації в meta_logs
4. **Статистика ефективності** роботи в офлайн режимі

### Для системи
1. **Зменшення стресу** на мережеві ресурси через фонова синхронізація
2. **Покращена UX** з оптимістичними оновленнями
3. **Надійність** при нестабільному інтернет-з'єднанні
4. **Масштабованість** завдяки локальному збереженню стану

## 📊 Метрики успіху

### Кількісні показники
- **Зменшення блокуючих помилок**: 90%+ зменшення скарг на мережеві проблеми
- **Час відновлення роботи**: < 5 секунд після відновлення інтернету
- **Успішність синхронізації**: > 95% автоматичних повторних спроб

### Якісні показники
- **Задоволеність користувачів**: Опитування показує покращення UX
- **Продуктивність**: Зменшення часу на повторну збірку замовлень
- **Надійність**: Менше ручних інтервенцій адміністраторів

## ⚠️ Ризики та міркування

### Технічні ризики
1. **LocalStorage обмеження**: 5-10MB на домен, потрібно очищення застарілих даних
2. **Service Worker підтримка**: Не всі браузери підтримують SW, fallback на звичайні retry
3. **IndexedDB складність**: Потрібно обробляти помилки запису/читання

### Бізнес-ризики
1. **Дублікація даних**: Можливі конфлікти між локальним та серверним станом
2. **Безпека**: LocalStorage доступний JavaScript, чутливі дані не зберігати
3. **Синхронізація**: Можливі race conditions при паралельних оновленнях

### План міграції
1. **Поступове впровадження**: Почати з localStorage, потім додати оптимістичні оновлення
2. **A/B тестування**: Тестувати на невеликій групі користувачів перед повним запуском
3. **Rollback план**: Можливість відключити офлайн-функціонал через feature flag
