# План рефакторингу `orders.payloadData` для якісної аналітики та точного логування комплектації

## 🔍 Поточна ситуація

У полі `orders.payloadData` зараз зберігається змішаний технічний формат, який добре працює для поточного списання залишків, але погано підходить для аналітики та розбору реальної логіки комплектації.

### Що є зараз

- `payloadData.shipment.bySku` містить спрощений мапінг `SKU -> { accGood, quantity }`.
- Для монолітних комплектів `accGood` дублюється для кожного SKU, хоча це не є корисною аналітичною ознакою.
- Коробки, hard cap, звичайні товари, монолітні комплекти та ручні рішення комплектувальника фактично живуть у різних частинах UI-логіки, але не мають спільного канонічного журналу.

### Чому це проблема

1. **Аналітика не бачить реальний процес комплектації**
   - Зараз видно лише підсумок по SKU.
   - Не видно, у яку коробку товар потрапив, чи був це монолітний набір, чи legacy-варіант, чи ручне відхилення від рекомендованого сценарію.

2. **Зайві дублікати технічних полів**
   - `accGood` повторюється на рівні кожного SKU, хоча для монолітних комплектів це, по суті, спільний атрибут.
   - Це створює шум у даних і ускладнює подальший аналіз.

3. **Змішані відповідальності**
   - Один і той самий JSON одночасно використовується для списання залишків, експорту в Dilovod і звітності.
   - Через це будь-яка зміна структури ризикує зламати кілька незалежних сценаріїв.

4. **Недостатньо прозорі бізнес-сценарії**
   - Потрібно окремо фіксувати сценарії, коли:
     - монолітний комплект зібраний як монолітний;
     - комплект навмисно переведено в legacy/regular формат;
     - коробка містить суміш монолітних і звичайних позицій;
     - частина товарів не влізла через hard cap.

---

## 🎯 Цільова модель

Для `orders.payloadData` пропонується перейти до **двошарового формату**:

1. **`shipmentSummary`** — компактний канонічний шар для обліку, списання залишків і базової аналітики.
2. **`packing` / `Комплектування`** — операційний шар, який фіксує, як товар був зібраний у процесі підготовки замовлення: монолітно, розкладено, вручну переведено в legacy-режим чи залишено без пакувального рішення.
3. **`packingManifest`** — деталізований журнал того, як саме замовлення було зібрано по коробках і чому були прийняті ті чи інші рішення.

### Принципи

- Не дублювати invariant-поля в кожному SKU без потреби.
- Не змішувати обліковий summary з операційним журналом комплектації.
- Зберігати legacy/forced-regular/monolithic як явні режими або рішення, а не як неявні побічні ефекти структури.
- Робити схему сумісною зі старими записами через адаптер читання.

## 📦 Пропонований формат

### 1. `shipmentSummary`

Це мінімальний шар, який відповідає на питання: що саме було відвантажено в обліковому сенсі.

Приблизна форма:

```json
{
  "shipmentSummary": {
    "version": 1,
    "mode": "mixed",
    "totals": {
      "regularItems": 3,
      "monolithicItems": 7,
      "boxes": 3
    },
    "bySku": {
      "02022": {
        "quantity": 1,
        "itemMode": "monolithic"
      },
      "07091": {
        "quantity": 1,
        "itemMode": "monolithic"
      },
      "02028": {
        "quantity": 4,
        "itemMode": "legacy_regular"
      }
    }
  }
}
```

### 2. `packingManifest`

Це детальний шар для аудиту, відтворення процесу та майбутньої аналітики.

Приблизна форма:

```json
{
  "packingManifest": {
    "version": 1,
    "createdAt": "2026-06-26T10:00:00.000Z",
    "packingMode": "mixed",
    "boxes": [
      {
        "boxNum": "1",
        "boxSize": "NK",
        "boxHCap": 24,
        "usedCapacity": 20,
        "lines": [
          {
            "sku": "07122",
            "quantity": 1,
            "lineType": "monolithic_set",
            "mode": "monolithic",
            "portionsPerItem": 20,
            "components": [
              { "code": "03009", "qty": 4 },
              { "code": "03028", "qty": 4 }
            ]
          }
        ]
      }
    ],
    "decisions": [
      {
        "sku": "07091",
        "decision": "forced_regular",
        "reason": "operator_override"
      }
    ]
  }
}
```

### 3. Metadata

Додатково варто мати невеликий мета-блок:

- `schemaVersion`
- `source` (`orderView`, `manualAdjustment`, `migration`, `legacyAdapter`)
- `createdAt`
- `createdBy`
- `packingMode`
- `compatibility` для старих записів

### 4. `packing` / `Комплектування`

Це проміжний шар між summary та manifest. Він потрібен для того, щоб не втрачати операційну логіку зборки, але й не роздувати summary технічними деталями.

Приблизна форма:

```json
{
  "packing": {
    "version": 1,
    "mode": "mixed",
    "items": [
      {
        "sku": "07104",
        "quantity": 1,
        "decision": "monolithic"
      },
      {
        "sku": "02028",
        "quantity": 4,
        "decision": "legacy_regular"
      }
    ]
  }
}
```

Призначення цього шару:

- показати, як саме комплектувальник або UI-логіка інтерпретували товар перед фінальним manifest;
- бути джерелом для подальшого audit trail;
- зберегти явне рішення про monolithic / legacy / regular без прив'язки до коробок.

## 🧩 Бізнес-правила

### Монолітні набори

- Якщо товар зібрано як монолітний, це має бути видно в manifest і в summary.
- Якщо товар пройшов через шар `Комплектування`, це рішення має бути відображене окремо від summary і manifest.
- Для монолітного комплекту зберігається оригінальна кількість комплектів, а не розгорнуті компоненти.
- `portionsPerItem` або аналогічна похідна ознака може бути корисною для розрахунку обсягу, але не повинна замінювати реальну кількість комплектів.

### Legacy/regular формат

- Якщо комплектувальник свідомо обрав старий формат, це має бути зафіксовано як окреме рішення.
- Таке рішення не повинно маскуватися під monolithic-режим.

### Коробки та hard cap

- Коробка є операційною одиницею логування.
- Для кожної коробки потрібно знати:
  - її номер;
  - розмір;
  - hard cap;
  - фактично використану ємність;
  - список ліній комплектації;
  - що залишилося поза коробкою через ліміт.

### Вкладені компоненти

- Якщо комплект містить вкладені компоненти, це має бути видно в packingManifest.
- Для summary вкладеність не повинна роздувати модель, якщо вона не потрібна для списання.

---

## 🔧 Які поля треба уникати або обмежити

- Не дублювати `accGood` у кожному SKU, якщо він не є специфічним винятком.
- Не зберігати в summary коробкову топологію або UI-стан.
- Не кодувати логіку legacy/monolithic через неочевидні порожні або спеціальні поля.
- Не змішувати в одному й тому ж рівні дані для обліку і дані для відтворення процесу пакування.

## 🛠️ План реалізації

### Фаза 1. Формалізація схеми

1. Зафіксувати контракт `shipmentSummary`.
2. Зафіксувати контракт `packingManifest`.
3. Описати допустимі `packingMode` та `decision` значення.
4. Визначити `schemaVersion`.

### Фаза 2. Адаптер генерації

1. Винести побудову summary з UI-стану в окремий хелпер або service.
2. Побудувати manifest на основі `expandedItems`, `boxes` і рішень комплектувальника.
3. Додати явну обробку monolithic / legacy / forced-regular сценаріїв.

### Фаза 3. Серверне споживання

1. Оновити читання `payloadData` на сервері через адаптер сумісності.
2. Списання залишків має дивитися лише на summary-шар.
3. Звітність може використовувати manifest для деталізації, але не повинна залежати від UI-структури.

### Фаза 4. Міграція і сумісність

1. Старі `payloadData` без нових секцій повинні автоматично адаптуватися.
2. Нові записи можуть зберігати обидва шари одночасно.
3. Протягом перехідного періоду допускається dual-read.
4. Після стабілізації можна поступово переводити споживачів на новий контракт.

---

## 📊 Що це дасть

- Якіснішу аналітику по фактичній комплектації.
- Менше шуму в даних завдяки винесенню технічних інваріантів з SKU-рівня.
- Можливість чесно відрізняти monolithic, legacy і forced-regular сценарії.
- Прозорий аудит: чому саме замовлення було зібрано певним чином.
- Простішу еволюцію схеми без ламання поточного списання залишків.

---

## ⚠️ Ризики

1. **Подвійна модель даних**
   - Потрібно чітко розділити, який шар за що відповідає.

2. **Сумісність зі старими записами**
   - Старий flat-формат ще якийсь час доведеться підтримувати.

3. **Ризик надмірної деталізації**
   - Якщо manifest стане занадто великим, ним буде незручно користуватися в аналітиці.

4. **Ручні винятки**
   - Якщо operator override не фіксувати явно, нова схема знову втратить пояснюваність.
