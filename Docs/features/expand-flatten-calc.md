# Серверне агрегірування `calc` та коротка інструкція

Коротко про зміни, які впливають на розгортання наборів та відображення коефіцієнтів у UI.

- Мета: зменшити мережевий шум та latency при розгортанні наборів, і забезпечити коректний показ `displayRatio` у `OrderChecklistItem`.
- Ключове: серверний ендпоінт `POST /api/expand/flatten` повертає для кожного продукту поле `calc` з агрегованими значеннями — `sumPortionsOne` та `weightKgOne`.

Що змінено (коротко)
- `server/routes/expand.ts` — batch-ендпоінт повертає precomputed `calc` для продуктів.
- `client/lib/orderAssemblyUtils.ts`
  - batch-підвантаження продуктів через `/api/expand/flatten` і пер-навантаження кеша `prod:SKU`.
  - `computeFlattenedComponent()` тепер читає `product.calc` і кешує результат під `calc:SKU`.
  - `addOrUpdateExpandedItem()` тепер проксіює `product.calc` та `product.unitRatio` у запис `expandedItems`.
  - додана нормалізація одиниць ваги (kg → g при підозрілими значеннями) та агрегована інструментація запитів.
- `client/components/OrderChecklistItem.tsx` — UI віддає пріоритет `item.calc.sumPortionsOne` при розрахунку `displayRatio` для звичайних товарів.

Коротка інструкція перевірки
- Виклик: `POST /api/expand/flatten` з набором SKU (наприклад, SKU з замовлення №17828) — переконайтеся, що відповідь містить `products[SKU].calc`.
- В UI: відкрити `OrderView` для того ж замовлення — бейдж `displayRatio` має показувати значення на основі `calc.sumPortionsOne` (наприклад 1.25, 0.75 і т.д.).

Причини та ефекти
- Менше індивідуальних GET `/api/products/:sku` у розгортанні → менше затримок і шумних логів.
- Централізоване агрегування дозволяє стабільніші результати при складних вкладених наборах.

Якщо потрібно — розширю документ відповідно до CI процесу або додати приклади запитів/очікуваних відповідей.
